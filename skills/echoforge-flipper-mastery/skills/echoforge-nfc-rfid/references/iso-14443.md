# ISO-14443 A/B Framing

ISO/IEC 14443 is the foundational standard for proximity cards at 13.56 MHz. Two incompatible initialization procedures coexist (Types A and B); both settle into the T=CL APDU transport of ISO-14443-4 for smartcard-ish operations.

Flipper's ST25R3916 driver handles anti-collision and activation. This document is what you need to know when a `.nfc` file or capture behaves weirdly.

---

## Part -3 (initialization & anti-collision)

### Type A (MIFARE, NTAG, most access cards)

1. **Reader REQA** (0x26) — "any cards in field?"
2. **Card → ATQA** (2 bytes) — "yes, I'm here; bit-frame anti-collision, UID size X." ATQA encodes UID size (4/7/10 bytes) and whether bit-frame or 16-slot anti-collision.
3. **Reader SEL_CL1** (0x93 ...) — "select cascade level 1 of the UID."
4. **Card → UID0..UID3 + BCC** — UID chunk. If UID > 4 bytes, first byte is `0x88` (Cascade Tag) and real bytes follow across higher cascade levels.
5. **Reader SEL completes per cascade** — up to 3 cascades (4/7/10-byte UIDs).
6. **Card → SAK** (Select Acknowledge, 1 byte) — tells reader what kind of tag this is:
   - `0x08` — MIFARE Classic 1K
   - `0x18` — MIFARE Classic 4K
   - `0x00` — MIFARE Ultralight / NTAG
   - `0x20` — ISO-14443-4 (APDU-capable, i.e. DESFire, JavaCard)
   - `0x28` — MIFARE Plus
7. **If SAK says ISO-14443-4:** reader sends RATS, card responds ATS, negotiate FSDI/FSCI + CID + speed; then APDU phase.

### Type B (passports, some Euro payment)

1. Reader **REQB** (0x05 AFI ...) — anti-collision by slot.
2. Card → ATQB (12 bytes) including PUPI (pseudo-UID), protocol type, TR0/TR1 timing.
3. Reader ATTRIB (0x1D) — assigns CID, finalizes activation.
4. Direct to APDU phase.

Type B is rarer in access control; common in government ID (ePassport, some national ID), some transit, and banking cards outside North America.

## UID sizes — what bytes mean

| UID size | Typical | Structure |
|----------|---------|-----------|
| **4 bytes** | MIFARE Classic 1K/4K, old cards | Random or semi-sequential ID; not globally unique (collisions exist) |
| **7 bytes** | NTAG21x, Ultralight, DESFire, MIFARE Plus | First byte is **NXP manufacturer code `0x04`**; following 6 bytes from NXP's fab pool, unique |
| **10 bytes** | Rare (some JavaCard) | Three cascade levels |

**Why UID size matters for cloning.** Many readers key access decisions on UID alone. A "write-UID" MIFARE Classic clone is a special backdoor tag (`FUID`, `CUID`, `UFUID` — see `mifare-classic.md`) that lets you program block 0 (which normally holds a fixed UID). NTAG21x / Ultralight have no such backdoor — their UID is locked.

## Block / frame structure

### Short frame (7 bits)

Used only for REQA/WUPA: 7-bit command, no CRC.

### Standard frame

```
[SOF] [DATA bytes...] [CRC 2 bytes] [EOF]
```

CRC is CRC-A polynomial `0x1021`, init `0x6363`. Flipper computes this automatically in the `.nfc` save; hand-crafted frames in RAW mode need you to compute it yourself.

### Bit-oriented frame (anti-collision)

Variable-length partial UIDs during collision resolution. Looks like a normal frame minus the CRC, with a bit-count + byte-count prefix.

## APDU phase (ISO-14443-4)

Once activated as "4-compatible" (SAK 0x20 / 0x28 etc.), cards speak APDUs:

```
CLA INS P1 P2 [Lc [Data]] [Le]
```

- **CLA** — class byte. `0x00` ISO, `0x90` proprietary, `0x80` proprietary.
- **INS** — instruction (0xA4 = SELECT, 0xB0 = READ BINARY, 0xD6 = UPDATE BINARY, 0x88 = INTERNAL AUTH, etc.).
- **P1/P2** — parameters.
- **Lc** — data length (1 byte), then **Data**.
- **Le** — expected response length.

Card responds with `[ResponseData] SW1 SW2`. SW `9000` = success; `6A82` = file not found; `6986` = command not allowed; `6982` = security status not satisfied.

Flipper's `nfc_relay` app and `nfc_magic` manipulate at this level. DESFire uses a proprietary CLA `0x90` for most commands.

## Common capture interpretations

**UID only (no data):** Reader issued REQA, did anti-collision, and either moved on or couldn't auth. You have the UID but no access to content. Still useful for cloning UID-only readers.

**UID + first sector of MIFARE Classic:** Reader authed with default key `FFFFFFFFFFFF` (still depressingly common). You can dump sector 0 contents.

**UID + all sectors:** Reader had real keys; dump is complete. Diversified-key deployments may even give per-card keys in block 0's manufacturer data — check for patterns.

**UID + ATS + APDU exchanges (DESFire):** Reader spoke ISO-14443-4 directly. The APDUs contain cmds + encrypted-responses; without the key you can replay a single session but not synthesize new ones.

## Relevant Flipper file fields

A `.nfc` header for Type A:

```
Filetype: Flipper NFC device
Version: 4
Device type: Mifare Classic 1k
UID: 04 A1 B2 C3
ATQA: 00 04
SAK: 08
```

For DESFire:
```
Filetype: Flipper NFC device
Version: 4
Device type: Mifare DESFire
UID: 04 A1 B2 C3 D4 E5 F6
ATQA: 03 44
SAK: 20
ATS: 06 75 77 81 02 80
```

The ATS byte sequence after the SAK is how you tell DESFire from "any 14443-4 card". DESFire's first ATS byte is usually `0x06-0x0A`.
